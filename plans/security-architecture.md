AI Agent Bridge Security Architecture

Overview

AI Agent Bridge enables remote SDKs and human operators to securely communicate
with agent sessions running on a login user's machine — a Linux desktop or
server, a macOS workstation, a cloud VM, or a Kubernetes workload with a
virtual display.

The bridge process runs as the login user inside an interactive session,
inheriting the user's display environment, credentials, and home directory.
This is a deliberate design requirement: AI agent CLIs (claude, codex,
opencode, gemini) are user-space programs that need access to `$DISPLAY`,
`$WAYLAND_DISPLAY`, `$XDG_RUNTIME_DIR`, `~/.claude`, API keys, and on
graphically-capable machines, running browser and emulator processes.

The design follows a practical Zero Trust model:

* The network is not trusted.
* Every client and server has a cryptographic identity.
* Mutual TLS (mTLS) is used for all remote SDK-to-server communication.
* Human operators authenticate through OIDC when Step CA is available, or
  through pre-issued client certificates in simpler deployments.
* Automated SDK clients authenticate through either OIDC or secure bootstrap
  tokens retrieved from a secret store.
* All certificates are issued by a private Certificate Authority (CA), either
  auto-generated by bridgectl or provided by a Step CA instance.
* Local access (unix socket, same machine, same user) requires no
  authentication — the filesystem permission model is sufficient.

Security Goals

Authentication

Verify the identity of:

* Remote SDK clients (automated processes)
* Human operators connecting remotely to observe or interject in sessions
* The bridge server itself (server certificate)

Confidentiality

Protect all remote traffic from interception.

Integrity

Prevent modification of messages in transit.

Authorization

Control access based on authenticated identity, not IP address.
For session write access specifically: enforce the active-writer protocol so
only the designated client can send input to the agent at any time.

Operational Simplicity

* Local use (single user, unix socket) requires no PKI configuration.
* Remote use auto-generates PKI on first `--listen` start.
* Step CA integration is optional — it upgrades the trust model for teams that
  already run OIDC infrastructure.

⸻

Threat Model

Assumptions

AI Agent Bridge is reachable only through a WireGuard or Tailscale VPN.

The bridge server MUST NOT be exposed directly to the public Internet.

Example:

Internet
    │
    ├── blocked
    │
VPN
    │
    ├── dev-workstation (bridgectl --listen 10.x.x.x:9445)
    ├── build-server    (bridgectl --listen 10.x.x.y:9445)
    └── ci-runner

Network access alone does not grant application access.
All remote application-level access requires certificate authentication.

Local access (unix socket, same machine) is restricted to the same user by
filesystem permissions on the socket file.

Out of Scope

* Protecting the machine itself from a compromised login user — the bridge
  process runs as that user and inherits their trust boundary
* Preventing the agent process from taking actions the user is authorised to
  take (e.g. writing files, running commands) — policy for agent behaviour is
  the responsibility of the agent CLI and operator configuration
* Multi-user machines where different users share one bridge instance — each
  user runs their own bridgectl server

Without Step CA, bridgectl auto-generates a self-signed CA and issues its
own server and client certificates. This covers the majority of developer and
homelab deployments.

With Step CA, certificate issuance is delegated to the CA and OIDC is used to
authenticate both human and automated clients before certificates are issued.

Certificate Authority

Two tiers are supported. Operators choose based on their infrastructure.

Tier 1: Auto-Generated PKI (default)

On first `bridgectl server start --listen`, the server generates:

    ~/.ai-agent-bridge/certs/
    ├── ca.crt              ECDSA P-384 root CA (10-year validity)
    ├── ca.key              CA private key (mode 0600)
    ├── server.crt          Server certificate (90-day validity)
    ├── server.key          Server private key (mode 0600)
    ├── local-client.crt    CLI's own client cert (for health probes)
    ├── local-client.key
    ├── jwt-signing.key     Ed25519 JWT signing key for local CLI
    ├── jwt-signing.pub
    ├── ca-bundle.crt       Trust bundle (CA cert)
    └── clients/
        └── <name>/
            ├── <name>.crt  Per-client certificate
            ├── <name>.key
            └── jwt-signing.key

The CA is self-signed and trusted only by clients that have received the
ca-bundle.crt out of band (e.g. via `bridgectl server issue-client`).

Per-client credentials are issued with `bridgectl server issue-client --name <name>`.
Each client gets its own certificate and JWT signing key so credentials can be
rotated or revoked independently.

Tier 2: Step CA Integration (optional)

For teams with existing OIDC infrastructure, Step CA can be used instead of
the auto-generated CA. Configure with:

    bridgectl server start \
      --listen 10.x.x.x:9445 \
      --step-ca-url https://step-ca.internal:443 \
      --step-ca-root /etc/step/root_ca.crt

When Step CA is configured:

* The auto-generate PKI step is skipped.
* Server certificate is obtained from Step CA at startup and renewed automatically.
* Human operators authenticate via OIDC → Step CA → short-lived client cert.
* Automated SDK clients use enrollment tokens retrieved from a secret store.

Step CA SHOULD only be accessible from the VPN.

⸻

Access Modes

Mode 1: Local (unix socket, no auth)

Default when `--listen` is not set.

The server listens on `~/.ai-agent-bridge/server.sock`. Only processes running
as the same user on the same machine can reach it (filesystem permissions).
No TLS, no JWT. The gRPC passthrough interceptor injects anonymous claims so
RPCs function.

Suitable for: local development, bridgectl run, IDE integrations.

Mode 2: Remote (TCP, mTLS + JWT)

Enabled when `--listen <addr>` is set.

The server binds to the given address. All connections require:

1. A valid client certificate issued by the trusted CA bundle.
2. A valid JWT signed by the per-client Ed25519 signing key.

Suitable for: SDK clients on other machines, human operators over VPN,
CI/CD pipelines connecting over a private network.

⸻

SDK Client Authentication

Automated SDK clients (orchestrators, control planes, CI runners) authenticate
using pre-issued credentials.

Flow (Tier 1 — auto PKI)

Operator runs on the bridge host:

    bridgectl server issue-client --name ci-runner

This produces four files:

    ca-bundle.crt      — trust bundle for verifying the server
    ci-runner.crt      — client certificate
    ci-runner.key      — client private key
    jwt-signing.key    — Ed25519 signing key for JWT tokens

The operator copies these to the SDK host. The SDK uses them for every
connection:

    client, _ := bridgeclient.New(
        bridgeclient.WithTarget("10.x.x.x:9445"),
        bridgeclient.WithMTLS(bridgeclient.MTLSConfig{
            CABundlePath: "ca-bundle.crt",
            CertPath:     "ci-runner.crt",
            KeyPath:      "ci-runner.key",
            ServerName:   "server",
        }),
        bridgeclient.WithJWT(bridgeclient.JWTConfig{
            PrivateKeyPath: "jwt-signing.key",
            Issuer:         "ci-runner",
            Audience:       "bridge",
            TTL:            5 * time.Minute,
        }),
    )

Flow (Tier 2 — Step CA)

Server Startup
      │
      ▼
Join VPN
      │
      ▼
Retrieve Enrollment Token from Secret Store
(Vault, AWS Secrets Manager, 1Password, SOPS)
      │
      ▼
Request Certificate from Step CA
(using enrollment token as one-time provisioner credential)
      │
      ▼
Receive Short-Lived Client Certificate
      │
      ▼
Connect to Bridge Server

Each client MUST have a unique enrollment token.

Shared enrollment tokens MUST NOT be used.

⸻

Human Operator Authentication

Human operators need to observe and optionally take over running agent sessions.
They authenticate interactively.

Flow (Tier 1 — pre-issued cert)

The bridge operator issues a credential for the human:

    bridgectl server issue-client --name mark

The human copies the four files to their machine and runs:

    bridgectl session attach --take-over <session-id>

bridgectl uses the pre-issued cert and JWT key automatically when connecting
to a secure-mode server (it reads `~/.ai-agent-bridge/certs/` on the local
machine, which the human populated with the issued files).

Flow (Tier 2 — Step CA + OIDC)

Human Startup
      │
      ▼
OIDC Login (Google / GitHub / Okta / Auth0 / Entra)
      │
      ▼
Step CA validates OIDC token
      │
      ▼
Short-Lived Client Certificate Issued (24h or 7d)
      │
      ▼
bridgectl session attach --take-over <session-id>

Example OIDC Identity

mark@company.com

or

CN=mark-laptop
SAN=mark@company.com

Certificate Lifetime

Tier 1: 90-day cert, rotated by re-running issue-client
Tier 2: 24 hours for humans, renewable automatically

⸻

Mutual TLS

All remote (TCP) connections MUST use mTLS.

Both sides authenticate each other.

Client Validation

Bridge server validates:

Certificate issued by trusted CA bundle
Certificate not expired
JWT valid: correct audience, issuer known, not expired, within MaxTTL

Server Validation

SDK / human client validates:

Certificate issued by trusted CA bundle
Certificate not expired
ServerName matches configured server name ("server" by default in auto-PKI;
or the FQDN when Step CA issues the cert)

⸻

Session Write Authorization

In addition to connection-level mTLS + JWT auth, session write access is
enforced at the application layer via the active-writer protocol.

Only the client holding the active writer slot may call WriteInput or
ResizeSession. Any other authenticated client that calls WriteInput receives
PERMISSION_DENIED.

The active writer slot is acquired by:

* StartSession: the caller becomes the active writer automatically
* AttachSession with role=ACTIVE: claims the slot if unoccupied
* ClaimWriter with force=false: claims the slot if unoccupied
* ClaimWriter with force=true: forcibly takes the slot from the current holder

The slot is released by:

* ReleaseWriter RPC
* Ctrl-] (detach key) in bridgectl session attach
* Client disconnect (stream closed by network event)

Observer clients (AttachSession with role=OBSERVER) receive all session output
but cannot write. Any number of observers may be attached simultaneously.

When the writer slot changes, a WRITER_CLAIMED or WRITER_RELEASED event is
broadcast to all observers (including the previous holder).

⸻

Authorization

Authorization is identity-based.

Authorization MUST NOT rely on:

* Source IP
* VPN IP
* Hostname
* DNS records

Policy applied at the server:

users:
  mark@company.com:
    role: admin         # can start/stop any session, claim any writer slot
  ci-runner:
    role: automation    # can start sessions, write to owned sessions only
  readonly:
    role: observer      # can only attach as OBSERVER

The server evaluates the JWT `sub` and certificate CN to determine the
caller's role. Role-based policy enforcement is a Tier 2 / Step CA feature;
in auto-PKI mode, any valid certificate holder has full access.

⸻

Certificate Renewal

Tier 1 (auto-PKI): server and client certificates have 90-day validity.
Renewal is manual: re-run `bridgectl server start --listen` to renew the
server cert (EnsurePKI renews if cert is within 30 days of expiry);
re-run `bridgectl server issue-client --name <name>` to renew client certs.

Tier 2 (Step CA): renewal is automatic.

Recommended parameters:

Lifetime: 30 days
Renew after: 20 days

If Step CA is temporarily unavailable:

* Existing certificates continue to function
* Existing connections continue to function
* New connections work until certificates expire
* Only renewal requires Step CA availability

⸻

Revocation

Tier 1: revoke a client by deleting its JWT public key from
`~/.ai-agent-bridge/certs/clients/<name>/` and restarting the server. The
removed key is no longer loaded; subsequent tokens from that client are
rejected.

Tier 2: revoke via Step CA + OIDC:

Compromised credentials: disable the OIDC account
Compromised certificate: revoke in Step CA
Terminated operator: remove from OIDC group and revoke cert
Compromised SDK client: delete the enrollment token from the secret store

⸻

Secure Storage Requirements

Private keys MUST remain local.

Private keys MUST NEVER be:

* Checked into Git
* Stored in source code
* Shared between machines

All keys are generated in ~/.ai-agent-bridge/certs/ with mode 0600.

The server checks key permissions at startup and logs a warning if permissions
are too permissive. (Enforcement of chmod is the operator's responsibility.)

Recommended backup: export the CA key offline and store it in a hardware key
store or encrypted offline medium. The server cert and client certs can be
re-issued from the CA at any time.

⸻

Network Security

WireGuard or Tailscale provides:

* Private transport
* Encryption
* Host reachability

VPN membership alone MUST NOT grant application access.

The VPN serves only as a network transport layer.

Authentication and authorization remain certificate-based at the application
layer.

bridgectl MUST only bind `--listen` to the VPN interface address (e.g.
`10.x.x.x:9445`), not `0.0.0.0:9445`, unless the operator explicitly
understands the exposure. Binding to all interfaces on a cloud VM exposes
the server to the internet.

⸻

Security Summary

AI Agent Bridge uses a layered security model:

1. WireGuard or Tailscale restricts network access to authorised participants.
2. mTLS authenticates every remote connection — both client and server present
   certificates.
3. JWT verifies per-RPC claims — issuer, audience, expiry, and max TTL.
4. Auto-generated PKI covers the default case; Step CA + OIDC covers teams
   with existing identity infrastructure.
5. Session write access is enforced at the application layer via the
   active-writer protocol, regardless of which authenticated client is calling.
6. Local access (unix socket) is protected by the operating system's
   filesystem permission model, not by TLS.

This architecture provides a practical Zero Trust model for developer
workstations, build servers, cloud VMs, and Kubernetes workloads while
remaining operationally simple — a developer can get a secure remote session
running with a single `bridgectl server start --listen` command and
`bridgectl server issue-client --name my-sdk` to provision a remote client.
